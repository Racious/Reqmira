# Reqmira 使用教學

> 給初次使用者的導覽：環境是什麼、各欄位用途，以及兩個可立即照做的示範流程。

---

## 一、核心概念

### 工作區（Workspace）
一個本機資料夾，對應一個專案。裡面放兩種東西：

```
workspace/
├─ environments/      # 環境檔（*.yaml）
│  ├─ local.yaml
│  ├─ dev.yaml
│  └─ demo.yaml
└─ collections/       # API 請求，可分資料夾
   ├─ auth/login.api.yaml
   ├─ users/get-user.api.yaml
   └─ demo/httpbin-with-vars.api.yaml
```

所有資料都是純文字 YAML，存在老爺自己的硬碟，可直接進 Git。

### 開啟與自動初始化工作區

點頂列「📂 開啟工作區」選資料夾後，Reqmira 會自動判斷：

| 選到的資料夾 | 行為 |
|---|---|
| 本身已是工作區（含 `collections/` 等） | 直接開啟 |
| 其下已有 `reqmira/` 工作區 | 自動指向 `reqmira/` 開啟 |
| 尚未初始化（一般專案 repo） | 頂列下方跳出提示：**「要在其下建立 `reqmira/` 工作區嗎？」** |

**用在自己的專案時**：直接選**專案根目錄** → 按提示的「建立」→ 會在專案下生成：

```
<你的專案>/reqmira/
├─ environments/local.yaml   # baseUrl 範例、token 留空
├─ collections/              # 空，新請求儲存時自動填
├─ flows/                    # 空
└─ .gitignore                # 排除 *.secret.yaml 機密環境檔
```

之後把 `reqmira/` 一起 commit 進專案 repo，團隊即可共享 API 定義。
（初始化只補缺、不覆蓋既有檔；再開同專案不會洗掉內容。）

### 環境（Environment）是什麼？
**「同一支 API，在不同階段有不同的網址與金鑰」**——這就是環境要解決的事。

- 開發時打 `http://localhost:8080`
- 測試時打 `https://dev.example.com`
- 正式時打另一個網址，token 也不同

與其每次手動改網址，Reqmira 讓老爺把「會變的值」抽成**變數**，寫在環境檔裡：

```yaml
# environments/local.yaml
name: local
variables:
  baseUrl: "http://localhost:8080"
  token: ""          # 機密：留空、不進 Git
  userId: "1001"
```

請求裡用 `{{變數名}}` 佔位：

```
GET {{baseUrl}}/api/users/{{userId}}
Authorization: Bearer {{token}}
```

送出時，Reqmira 會依**頂列目前選的環境**把 `{{baseUrl}}` 等代換成實際值。
切換環境（頂列右側下拉），同一支請求就打到不同伺服器——這就是環境的價值。

> 機密欄位（token、password）建議在環境檔中留空，由使用時填入，並排除於 Git。

## 二、介面與可填表單逐一說明

### 頂列
| 元件 | 用途 |
|---|---|
| 📂 開啟工作區 | 選擇 workspace 資料夾，載入其中的 collection 與環境 |
| ⟳ | 重新整理（**新增/修改 YAML 檔後，按這個才會重新載入**） |
| 環境 ▾ | 切換目前環境，決定 `{{變數}}` 代入哪一組值 |
| ＋ 新請求 | 建立一支空白請求 |

### 中欄：Request Editor（編輯與送出）
| 表單 | 說明 |
|---|---|
| **方法**（下拉） | GET / POST / PUT / PATCH / DELETE / HEAD / OPTIONS |
| **URL**（輸入框） | 請求網址，可用 `{{baseUrl}}/api/...`；按 Enter 直接送出 |
| **送出** | 經 Rust 端實際發送（無 CORS 限制） |
| **儲存** | 寫回 YAML；新請求會先問「儲存為」路徑 |
| **請求名稱 / id** | 顯示名稱與識別碼（id 也是預設檔名） |
| `env: local` 標籤 | 提示目前套用的環境 |

下方三個分頁：

- **Query**：網址問號後的參數（`?status=PAID&page=1`）。每列可勾選啟用、填 Key/Value，尾端會自動長出新空列，✕ 移除。
- **Headers**：HTTP 標頭（如 `Authorization`、`Content-Type`），操作同上。
- **Body**：先選型別（none / json / text / form），json/text 會出現文字框輸入內容。

### 右欄：Inspector（差異化主功能）
不需送出，**邊打邊分析**目前這支請求：
- **Path 變數**：抓出 URL 裡的 `{userId}`、`:id`、`{{var}}`
- **Query 參數**：推測每個參數的型別與格式（integer / boolean / email…）
- **Headers**：辨識 Bearer Token、是否用了變數
- **Body Schema 推測**：把 JSON body 拆成型別樹（含巢狀、陣列長度、必填）

### 下欄：Response Viewer（送出後）
- 頂部顯示 **狀態碼、耗時、大小、Content-Type**
- 分頁：
  - **Pretty**：JSON 樹狀檢視，點欄位名可複製其路徑（如 `data.items[0].id`）
  - **Schema**：回應資料的型別結構推測
  - **Raw**：原始回應文字
  - **Headers**：回應標頭

## 三、示範流程 A：第一次送出（最快上手）

1. 頂列點 **📂 開啟工作區**，選本專案的 `workspace/` 資料夾。
2. 左欄展開 **demo**，點 **Demo · httpbin GET**。
3. 中欄會看到 `GET https://httpbin.org/anything`，Query 分頁有 `hello`、`page`。
4. 按 **送出**。
5. 下欄出現 **200**、耗時、大小；**Pretty** 分頁可展開回應 JSON。
6. 切到 **Schema** 分頁，看 httpbin 回應的型別結構。

✅ 這支用絕對網址，任何環境都能跑（需連網）。

---

## 四、示範流程 B：體驗「環境變數」（重點）

1. 確認已開啟 `workspace/`；若剛新增檔案，先按頂列 **⟳** 重新整理。
2. 頂列「環境」下拉切到 **demo**。
   （demo 環境的 `baseUrl = https://httpbin.org`、`token = demo-token-123`、`userId = 42`）
3. 左欄 demo 資料夾點 **Demo · 環境變數示範**。
   它的設定是：
   ```
   GET {{baseUrl}}/anything/{{userId}}
   Authorization: Bearer {{token}}
   ?from=reqmira
   ```
4. 看右欄 **Inspector**：會抓到 `userId`、`token`、`baseUrl` 為變數。
5. 按 **送出**。
6. 回應裡（httpbin 會原樣回顯）可看到：
   - 實際網址變成 `https://httpbin.org/anything/42`
   - `Authorization` 標頭是 `Bearer demo-token-123`
   - `from=reqmira` 出現在 args

   → **三個 `{{變數}}` 都被 demo 環境的值代入了。**
7. 進階：把頂列切回 **local**（`baseUrl=http://localhost:8080`）再送一次，會因本機無此服務而連線失敗——這正說明「環境決定了請求打去哪裡」。

## 五、建立自己的請求

1. 頂列 **＋ 新請求** → 填方法、URL（可用 `{{baseUrl}}`）、Query/Headers/Body。
2. 邊填邊看右欄 Inspector 的拆解。
3. 按 **送出** 測試。
4. 按 **儲存** → 輸入路徑如 `collections/orders/create-order.api.yaml` → 確認。
5. 檔案即以 YAML 存入工作區，可進 Git 版控。

## 六、Flow 自動化（多步驟流程）

**Flow = 可重播的 API 劇本**：把多支 API 串成依序自動執行的流程，前一步的回應可抽值餵給後一步，每步可自動斷言。

### Flow 檔結構

放在 `workspace/flows/`、副檔名 `.flow.yaml`：

```yaml
version: 1
id: login-and-query
name: 登入並查詢使用者
description: 說明（可選）

steps:                       # 依序執行的步驟
  - name: 登入
    request: collections/auth/login.api.yaml   # 引用一支已存在的請求
    extract:                 # 從回應抽值 → runtime 變數
      token: "$.data.access_token"
    assert:                  # 自動檢查
      status: 200

  - name: 查詢使用者
    request: collections/users/get-user.api.yaml
    variables:               # 此步額外/覆寫的變數
      userId: "1001"
    assert:
      status: 200
      json.code: "0000"
```

### 三個關鍵欄位

| 欄位 | 語法 | 說明 |
|---|---|---|
| `request` | 相對 workspace 路徑 | 引用一支已存在的 `.api.yaml`，不重寫請求 |
| `extract` | `變數名: JSONPath` | 從回應取值存為 runtime 變數，後續以 `{{變數名}}` 取用。路徑如 `$.data.token`、`$.items[0].id` |
| `assert` | `檢查項: 期望值` | `status: 200` 查狀態碼；`json.code: "0000"` 查 body 欄位（支援巢狀路徑） |

### 變數合併優先序（後者覆蓋前者）

1. 環境變數（頂列選的環境）
2. runtime 變數（前面步驟 extract 出來的）
3. 步驟 `variables`（該步寫死的）

### 怎麼跑

1. flow 檔放進 `flows/`，頂列按 **⟳** 重新整理
2. 左欄 **Flows** 區塊點該 flow → 跳出模態
3. 按 **▶ 執行** → 逐步顯示狀態、耗時、抽取值、斷言通過與否，及總結

### 示範（可實跑）

環境切 **demo**，左欄 Flows 點 **Demo · httpbin 兩步流程** → 執行：第一步抽取 `echoedUrl` 並斷言 200，第二步斷言 `json.method = GET`，應顯示「✓ 全部通過 2/2」。

> 典型用途：冒煙測試（部署後驗主流程）、回歸驗證（改後端後重跑）、省去手動 token 串接。

## 常見問題

- **新增的 YAML 檔在左欄沒出現？** 按頂列 **⟳** 重新整理。
- **送出顯示連線失敗？** 多半是該環境的 `baseUrl` 指向的服務沒在跑，或未連網。
- **變數沒被代入（仍是 `{{xxx}}`）？** 表示目前環境沒有定義該變數；檢查頂列環境與對應的 `environments/*.yaml`。
- **選了專案資料夾卻跳出「要建立 reqmira/ 嗎」？** 表示該資料夾尚未初始化為工作區。按「建立」即會在其下生成 `reqmira/` 骨架；按「取消」則不動作。
