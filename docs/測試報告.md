## 四、手動驗收流程清單

> 於 `npm start` 開啟的視窗中，開啟本專案 `workspace/` 後依序驗收。

1. **基礎送出**：環境選 demo → Demo · httpbin GET → 送出 → 應見 200、耗時、Pretty JSON 樹。
2. **環境變數**：Demo · 環境變數示範 → 送出 → 回應中 `{{baseUrl}}/{{userId}}/{{token}}` 皆已代入。
3. **Inspector**：建立使用者 → 右欄 Body Schema 顯示巢狀型別、`roles: array(2)`、必填/選填。
4. **Codegen**：送出後 → Codegen 分頁 → TypeScript / Java DTO / Markdown 三種輸出、可複製。
5. **Diff**：送出 → 設為基準 → 再送一次 → Diff 分頁顯示綠/紅/黃差異。
6. **History**：左欄 History 顯示歷次紀錄，點擊回顧、可設為基準。
7. **Flow**：左欄 Flows → Demo · httpbin 兩步流程 → ▶ 執行 → 「✓ 全部通過 2/2」。

## 五、開發中遇到的問題與修復紀錄

| # | 問題 | 根因 | 修復 |
|---|---|---|---|
| 1 | 鷹架專案名為 `_scaffold` | `create-tauri-app` 暫存目錄名 | 全面改寫 package.json/Cargo.toml/tauri.conf.json 等為 Reqmira |
| 2 | `npm start` 報 **port 1420 已占用** | 預覽面板殘留的 dev server 把持埠 | 查出 PID 已死、埠無監聽，重新啟動即通；後續一次到位 |
| 3 | 前端建構失敗 **TS2345** | `analyze.ts` 中 `{ type: "mixed" }` 字面值未窄化為 `SchemaNode` | 明確標註 `const itemNodes: SchemaNode[]` |
| 4 | 過渡建置出現 `diff_json` **never used 警告** | 指令尚未註冊進 `lib.rs` 的中間版本 | 完成註冊後最終版本正常；確認執行中程序為註冊後版本 |
| 5 | **查回來有點卡** | ①回應被 JSON.parse 兩次 ②樹預設展開多層 ③大陣列全渲染 ④debug build 慢 ⑤每次重建 HTTP client | ①Schema 共用解析 ②收斂展開層級 ③大陣列截斷 ④dev profile opt-level=3 ⑤reqwest client 以 OnceLock 重用 |
| 6 | **`toTypeScript` 陣列根遺失巢狀 interface** | 非物件根提前 return 別名，丟棄已收集的 defs | 改為先算別名（順帶收集）再與巢狀定義一併輸出；**新增回歸測試** |
| 7 | `parse_method("WAT!!")` 測試失敗 | 測試假設錯誤——`!` 是合法 HTTP token 字元，HTTP 允許自訂方法 | 修正測試改用含空白的非法 method；確認程式行為正確（符合規範） |

> 其中 #6 是撰寫測試時**真正發現的程式 bug**並修復；#7 則是測試假設被程式糾正——兩者皆印證了補測試的價值。

## 六、已知測試缺口與後續建議

（依子代理覆蓋盤點，按優先序）

1. **元件測試**：`KeyValueEditor`、`SchemaTree` 等可用 Vitest + `@vue/test-utils` 補（DOM 層）。
2. **Flow 編排 run() 測試**：需 mock `api` invoke 層；目前以純函式（looseEq/getByPath）與手動驗收覆蓋。
3. **網路 e2e 去網路化**：以 `wiremock` 之類的 mock server 取代 httpbin，使 e2e 可進 CI 不依賴外網。
4. **models serde 往返**：可補 `RequestSpec` 預設值、camelCase 反序列化等測試。

## 七、結論

- **核心邏輯（後端發送/解析/diff、前端推測/產生/路徑）皆有自動化測試覆蓋並全數通過。**
- GUI 互動以手動驗收清單補足（受 Tauri 架構限制無法自動點擊）。
- 過程中發現並修復一個真實 bug（陣列根 codegen），另修正一處測試假設。
- 建構、型別檢查、前後端測試全綠，專案處於可持續開發的健康狀態。
