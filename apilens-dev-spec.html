ApiLens — 輕量 API 工作台

一份可直接據以開工的整合開發定稿。由天城（AMAGI）整合三份草案精華：以 ApiLens 為主體、補入 Vela 的架構論述、吸收 Agent Console 的流程與審核機制。

本稿來源標記：章節內以 VELA APILENS CONSOLE 標示主要參考來源，便於老爺回溯。

1. 專案願景

打造一款輕量、離線優先、Git 友善的桌面 API 工作台，取代 Postman 在日常工作中的角色。核心不在功能堆疊，而在四件事：

更便利

啟動快、體積小（Tauri 約 5–10MB）、介面直覺；集合以純文字存於本機，可進 Git 版控與 code review。

更好讀

把深層巢狀的 Request / Response 拆解成樹狀、表格、型別著色與路徑標示，「看懂資料」不再費力。

更自動化

從回應自動抽取變數、串接下一請求；定義 Flow 一鍵跑完整串 API 並產出報告。

更能沉澱

從 Request/Response 直接產出 Markdown 文件、DTO / Swagger 草稿，銜接 Spring Boot 工作流。
定位：不重做一個 Postman。ApiLens 聚焦老爺真正的痛點——「拆解、分析、自動化、文件化」，以此為差異化主軸，其餘功能夠用即可。

2. 目標與痛點 APILENS

主要解決工作上測試 API 時「能發送，但不好理解、難整理、難自動化」的問題。

需求面向目前常見問題ApiLens 目標
工具負擔Postman 功能完整但偏重、啟動慢、吃記憶體Tauri 殼層，輕量原生
雲端綁定需登入、資料上雲、有鎖定疑慮完全離線，資料存本機純文字
參數閱讀Query/Header/Body 分散，JSON 深層欄位不好看自動拆解 Path/Query/Header/Body 與型別
文件產出測試完還要另寫 Markdown / Swagger從 Request/Response 直接產文件草稿
自動化流程重跑與 Token 串接需額外設定Flow Runner、變數擷取與 CLI
版本控管Collection diff 不直覺或仰賴雲端YAML / JSON，可直接放進 Git

3. 產品定位 APILENS

核心定位：ApiLens 是一款本機優先、Git 友善、以參數拆解與 API 閱讀為核心的輕量 API 分析工作台。

Local-first

Collection、環境變數、歷史紀錄優先存本機，不依賴雲端。

Readable-first

把 API 拆成工程師易懂的區塊、表格、樹狀與文件視圖。

Git-friendly

以 YAML / JSON 為主資料格式，便於 commit、diff 與 review。

Automation-ready

第一版即預留 Flow Runner、CLI、報告輸出與 AI 整合能力。

不做什麼

第一階段不追求完整取代 Postman 的平台功能——雲端團隊協作、Public API Network、大型 Mock 平台、商業 API Portal。這些會讓專案太早變重。

4. 技術選型 VELAAPILENS

核心原則：輕量、繞過瀏覽器限制、老爺易維護。

層選擇理由
桌面殼層Tauri 2成品輕巧、記憶體佔用遠低於 Electron；用系統 WebView，不打包整個 Chromium
前端Vue 3（Composition API + TypeScript）輕量、響應式強；觀念與 Angular 相通，老爺過渡平滑
核心引擎Rust（Tauri command）HTTP 由 Rust 端發送，繞過 CORS；原生效能，做高速 deep-diff
HTTP 函式庫reqwestRust 生態成熟的 async HTTP client
狀態管理Pinia + Vue RouterVue 官方推薦，輕巧直覺
編輯器/檢視CodeMirror / Monaco · 虛擬列表 · JSON Tree大型回應流暢呈現
建構工具ViteTauri + Vue 預設，熱更新快
儲存格式YAML / JSON可讀、Git 可 diff、AI 友善（取代 TOML 草案，統一格式）
歷史紀錄SQLite（可選）大量送出紀錄的查詢效能
格式決議：Vela 草案原採 TOML，本定稿統一改採 YAML——它能自然承載巢狀 Request 結構、生態廣、AI 解析方便，且已是 ApiLens 草案的選擇。

6. 核心功能 APILENS

6.1 Request Builder

建立並發送常見 HTTP Request：Method、URL、Path Variable、Query、Header、Cookie、Auth、Body。

GETPOSTPUTPATCHDELETE JSON BodyForm DataBearer TokenBasic Auth

6.2 Request Inspector（差異化主功能）

自動將 URL 與 Body 拆解成可讀結構，確認參數來源、型別與必要性——這正中老爺「參數易拆解」的核心需求。

區塊範例分析內容
Path/api/users/{userId}/orders辨識 userId 為 path variable
Query?status=PAID&page=1推測 status 為 enum、page 為 number
HeadersAuthorization: Bearer {{token}}辨識 Bearer Token，提示可綁環境變數
Body{"items":[...]}解析 JSON tree、陣列欄位與深層型別

6.3 Response Analyzer + Diff

JSON Tree

樹狀結構，可展開收合、複製欄位路徑（data.items[0].id）、型別著色。

Schema 推測

依 response 推測欄位型別，產生 DTO / TypeScript Interface 草稿。

Diff 比對

Rust 端結構化 deep-diff：綠＝新增、紅＝刪除、黃＝變更。

比對情境：① 同一請求兩次回應（看 API 是否變了）；② 兩個環境（dev vs prod 參數差異）。

6.4 文件產生器

從 Request / Response 直接產生 Markdown API 文件，降低手動整理成本。

# POST /api/orders

## Request Body
| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| userId | number | 是 | 使用者 ID |
| items[].sku | string | 是 | 商品代號 |

9. 檔案儲存格式 (YAML) APILENS

不自創 DSL。採 YAML / JSON，便於閱讀、Git diff、AI 分析與後續轉換。保留 version 欄位以利日後 migration。

Request YAML 範例

version: 1
id: get-user
name: 查詢使用者
method: GET
url: "{{baseUrl}}/api/users/{{userId}}"

pathParams:
  userId: { value: "1001", type: number, required: true, description: 使用者 ID }

queryParams:
  includeRole: { value: true, type: boolean, required: false }

headers:
  Authorization: "Bearer {{token}}"
  Accept: "application/json"

# 回應抽取 → runtime 變數
extract:
  - from: "$.data.token"
    to: "authToken"

tests:
  - { expect: status, toBe: 200 }
  - { expect: json.code, toBe: "0000" }

docs:
  tags: [User, Query]

Environment YAML 範例

name: local
variables:
  baseUrl: "http://localhost:8080"
  token: ""        # 機密欄位，排除於 Git
  defaultUserId: "1001"

11. 自動化 · Flow APILENSCONSOLE

自動化是 ApiLens 超越一般 API Client 的關鍵。Flow 定義吸收 Agent Console 的 workflow 設計——含觸發條件、輸入欄位、逐步定義，比單純串接更完整。

Flow YAML 範例

version: 1
id: login-and-query
name: 登入並查詢使用者
description: 先登入取得 token，再帶 token 查詢使用者

triggers: [login, smoke-test, 回歸驗證]   # 可供日後 AI/CLI 匹配

inputs:
  - { id: account, label: 帳號, type: text, required: true }

steps:
  - name: Login
    request: collections/auth/login.api.yaml
    extract:
      authToken: "$.data.access_token"      # 抽取 → runtime 變數

  - name: Query User
    request: collections/users/get-user.api.yaml
    variables: { userId: 1001 }
    assert:
      status: 200
      json.code: "0000"

Token Extractor

登入後自動從 response 擷取 token，寫入 runtime 變數，注入後續請求。

Assertion

檢查 status、header、json path、response time，產出測試結果。

HTML Report

批次執行後產生可分享的測試報告。

CLI 預留指令

apilens run collections/users/get-user.api.yaml --env local
apilens run-flow flows/login-and-query.flow.yaml --env dev
apilens export-doc collections/users --format markdown
apilens analyze response.json --to dto-java

12. AI 友善與沉澱機制 APILENSCONSOLE

資料採純文字 YAML / JSON，易與 Codex / Claude / 本機 Agent 整合。第一階段先產生 prompt，不直接串模型，降低不穩定與成本。

功能說明第一階段做法
Prompt Export依目前 API 產生可貼給 AI 的分析 prompt先產文字
DTO 生成從 schema 產生 Java DTO 草稿本機模板，再交 AI 優化
Controller 草稿依 API 定義產 Spring Boot Controller 範例輸出 prompt / markdown
測試案例建議依參數列出正常、邊界、錯誤案例由 analyzer 產基礎案例

沉澱安全護欄（借鑑 Agent Console 審核分級）

未來若讓 AI 自動歸納可重用的 Request 範本、文件或測試案例，採分級審核——AI 負責建議，老爺批准後才寫入：

等級說明建議
Level 1AI 提出建議，使用者確認後寫入第一版推薦
Level 2低風險紀錄自動寫入（如 run 結果、耗時）MVP 後加入
Level 3AI 自動修改範本與流程不建議第一版啟用
指導原則：AI 負責建議，人類負責批准沉澱。這套護欄確保自動化不致悄悄累積錯誤資產。

15. 風險與對策 VELAAPILENS

風險對策
Rust 學習曲線（老爺主力為 Java/Angular）後端範圍刻意收斂——多為「收參數→發請求→回 JSON」樣板，天城提供完整骨架與註解，漸進熟悉
範圍過大（追 Postman 全功能）嚴守「拆解＋分析＋自動化」主軸，其餘夠用即止
HTTP 細節繁多（Auth/Cookie/Redirect/TLS/Multipart）分階段支援，v0.1 不全做
YAML schema 早期不穩保留 version 欄位與 schema validator，預留 migration
機密（token/密碼）明文存檔敏感值標記、排除 Git（.gitignore），日後接系統金鑰庫
AI 功能過早介入先做 Prompt Export，模型串接後置；沉澱採 Level 1 審核
Tauri 2 生態較新核心走穩定 API；不確定處先最小可行驗證
